# 开源前清洁清单

生成源（你自己的工作目录）里难免混着个人信息——路径、姓名、内部工具引用、协作平台链接。**推公开前必须剥干净**。这是词境自己走过的清单，照抄改用。

## 一、成品（html + 资产）

- [ ] **html 全文搜个人信息**：姓名、`/Users/<你>`、iCloud/云盘路径、协作平台（Notion/飞书…）链接、内部项目名
  ```bash
  grep -oni "你的名字\|/Users/\|icloud\|notion\|<内部项目名>" index.html | sort | uniq -c
  ```
- [ ] **词条备注里的工作流注释**：像"XX 点单补词""2026-XX 返修"这类过程记录，会显示在词卡上，剔成干净释义
- [ ] **图片元数据**：转 JPEG 时顺带剥掉 PNG 生成元数据（`sips -s format jpeg` 自带）
- [ ] **中文/云端 TTS 音频**：edge-tts 等微软/云端声线**不随成品分发**（再分发法律灰色），走浏览器 Web Speech 兜底，或换 Apache/MIT 授权的本地模型重生成

## 二、工作流（脚本 + 文档）

- [ ] **脚本里的绝对路径**：`/Users/<你>/...` → 占位符 `<xxx目录>` 或相对路径
- [ ] **venv / 工具路径**：`~/你的项目/venv` → 文档化成"你的 XX venv"，别硬编码
- [ ] **注释/docstring 里的人名与内部项目**：`(XX 设计)`、`抄自 ~/项目/xxx`、协作平台 ID 全清成通用表述
- [ ] **岗位/Agent 定义里的溯源链接**：指向你私有文档的路径删掉

## 三、终扫（一条命令兜底）

```bash
grep -rn "你的名字\|/Users/\|icloud\|obsidian\|notion\|<内部项目>" . \
  --include="*.py" --include="*.js" --include="*.md" --include="*.html" --include="*.json" \
  | grep -v node_modules
# 输出为空 = 干净
```

## 四、结构：生成源与成品区分离

**生成源不动，另建成品区**。你日常在生成源（含所有过程文件、私有备注）工作；成品区只放**剥离干净、对外发布**的版本。加了新场景后，从生成源重新同步到成品区再提交。这样过程脏数据永远不会误推公开。

> 词境的做法：生成源在 Obsidian vault，成品区在 `~/Projects/cijing-github/`（两个 git 仓：成品 + 工作流）。
